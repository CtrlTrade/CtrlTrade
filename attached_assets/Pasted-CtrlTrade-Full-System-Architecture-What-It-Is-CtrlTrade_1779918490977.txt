CtrlTrade — Full System Architecture
What It Is
CtrlTrade is a multi-tenant SaaS CRM platform purpose-built for UK trades businesses. It has two distinct user tiers living inside the same application:

Super Admin — the CtrlTrade team (Alan, Daniel). Manages all tenants, billing, plans, marketplace, compliance queue, and platform-wide settings.
Tenant (CRM user) — a trades business (e.g. a roofing company). Gets their own company workspace with staff, customers, jobs, quotes, invoices, and all CRM features.
Both tiers are served by the same React frontend (artifacts/roofing-crm), the same Express API (artifacts/api-server), and the same PostgreSQL database — isolated by companyId / tenantId at every data layer.

Deployment Topology
ctrltrade.co.uk             ← live production domain
  /                         → Marketing Site  (artifacts/marketing, Vite/React)
  /app                      → CRM Frontend    (artifacts/roofing-crm, Vite/React)
  /api                      → API Server      (artifacts/api-server, Express 5)
  /portal/:slug             → Customer Portal (same CRM frontend, portal routes)

All traffic routes through a shared reverse proxy. The API is a single Express process bundled with esbuild, serving from port 8080. Static frontends are served by Vite in dev and built to static files in production.

Authentication & Authorization
Login methods:

Replit OIDC — Google/social login via Replit's OpenID Connect proxy. Calls upsertUser() on every login. This is the primary path for Alan and Daniel (super admins) and tenant owners.
Credential login (POST /api/auth/login) — Email + bcrypt password for hardcoded team accounts (user-alon, user-howlett, user-daniel) and self-service registered tenants.
Mobile (POST /api/mobile-auth/credential-login) — Same credential check but returns a session cookie for the Expo app.
Sessions: Server-side, stored in the PostgreSQL sessions table, signed with SESSION_SECRET. Session ID delivered via sid HttpOnly cookie.

Super Admin detection:

isSuperAdmin = "true" column on users table
Set automatically at startup for known emails (enforced in seedDemoTenants.ts)
Set on every OIDC login via upsertUser() for SUPER_ADMIN_EMAILS
Set on every credential login for user-alon / user-daniel
Tenant resolution:

Every authenticated API request resolves the caller's company via companyResolver.ts
Chain: session.user.id → tenant_users.tenantId → company.linkedTenantId
Returns TenantNotFoundError → HTTP 403 if no company found (fail-closed)
Seat-based access:

Control Seat — Full CRM access (office/admin/management users)
Field Seat — Limited to: assigned jobs, calendar, customer details, documents, assigned vehicle
FieldSeatGuard component on frontend; requireControlSeat middleware on sensitive API routes
Role-based permissions:

Tenant admins can define custom roles with granular permission toggles
roles table stores JSON permission maps
staff.permissions stores granted permissions per staff member
can(key) helper in permissions context checks admin/super-admin bypass first

Key Business Logic Flows
Lead → Quote → Job → Invoice pipeline:

Lead created manually or via webhook import
Lead converted → Quote (POST /leads/:id/convert creates draft quote)
Quote sent to customer (PDF or portal link)
Customer accepts via portal or manually marked won → POST /quotes/:id/convert creates a Job
Job runs; photos, costs, checklists logged
Job marked complete → Invoice auto-generated
Invoice sent via portal; customer pays via Stripe or manually marked paid
Invoice optionally pushed to Xero via sync
Compliance expiry monitoring:

Nightly job (12-hour interval) scans compliance_documents for items expiring within 60 days
Creates compliance_notifications entries
Dashboard shows alert badge; email sent to company
Multi-tenant billing:

pricing_config holds master seat/plan prices
On tenant signup, a tenant_pricing_snapshots row is created locking in the price
MRR = snapshot seat price × seat count + active add-on revenue
Platform overview aggregates all tenant MRR for platform revenue dashboard
Infrastructure & Security
CORS: Allowlist from REPLIT_DOMAINS env in production; open in dev. Fail-closed (403) if no origins configured in prod.
CSP: No unsafe-inline or unsafe-eval. Strict directives.
IDOR protection: Every /:id sub-route verifies the record belongs to the caller's companyId before returning or mutating.
File uploads: Two-pass MIME validation (Content-Type header + magic bytes). Stored in Replit Object Storage.
Secrets encrypted at rest: Xero and Google OAuth tokens encrypted with AES-256-GCM using ENCRYPTION_KEY (64 hex chars). Server throws on startup if key is missing.
Readiness check: GET /api/readyz verifies DB connection, sessions table, ENCRYPTION_KEY, and SESSION_SECRET. Returns 503 if any are missing.
Audit trail: activity_log table records significant data changes with actor, timestamp, and before/after snapshot.
This is the complete picture — from the public marketing site and customer portal through to the tenant CRM and the super admin platform layer, all running on the same codebase, same API, and same PostgreSQL database.
